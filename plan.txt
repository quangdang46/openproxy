# OpenProxy — Agent-First CLI PLAN
 
> **Mục tiêu**: Biến OpenProxy thành "agent-native". Mọi thứ làm được trên Web Dashboard phải làm được qua **CLI một dòng lệnh**, có **JSON output ổn định** (`--robot`), idempotent, scriptable. Agent (Claude Code, Codex, Cursor, Cline, Devin…) là user hạng nhất; dev/web là user hạng hai.
 
---
 
## 0. TL;DR
 
- **Stack**: Tiếp tục dùng **Rust + clap** — CLI và server là **cùng một binary `openproxy`** (đã có sẵn). Không thêm runtime mới.
- **Một mental model duy nhất**: CLI **tích hợp trực tiếp** vào `DATA_DIR` (db.json + SQLite). Không có khái niệm online/offline cho user. Mọi lệnh state đều ghi/đọc DB trực tiếp.
- **Server tự động pick up thay đổi DB**: thêm file watcher (notify-rs) để server reload `ArcSwap<AppDb>` khi `db.json` thay đổi → CLI ghi DB là server thấy ngay, không cần IPC/API call.
- **Lệnh runtime** (`logs tail`, `usage stream`, `chat stream`, `mitm start/stop`, `tunnel start/stop`) — bản chất cần server đang chạy. Nếu server down, CLI fail với gợi ý rõ: `error: server not running. Start it: openproxy server start --detach`.
- **Output flags chuẩn cho mọi lệnh**:
  - `--robot` → pure JSON tới stdout, không banner, không màu, schema có version.
  - `--verbose` (`-v`, `-vv`) → tăng log tracing tới stderr.
  - `--quiet` (`-q`) → chỉ exit code + lỗi.
  - `--no-color`, `--color=auto|always|never` (auto-detect TTY).
- **Auth**: mặc định KHÔNG cần auth (CLI ghi DB trực tiếp). Chỉ khi remote-manage server ở máy khác (ít gặp) mới dùng `--url` + `--api-key`.
- **Phạm vi**: 18 nhóm lệnh top-level, **bao phủ 100%** các trang `/dashboard/*` và 90+ `/api/*` endpoints hiện có.
- **Roadmap**: 6 milestone, bắt đầu từ command surface chuẩn hoá → mở rộng dần → đóng băng schema v1.

## 2. Nguyên tắc thiết kế (Agent-First)
 
1. **Một lệnh = một intent rõ ràng.** Không dồn nhiều side-effect vào một lệnh.
2. **Idempotent by default.** Có `apply`/`upsert` thay vì chỉ `add`. `delete` không lỗi nếu không tồn tại (cờ `--strict` để bắt lỗi).
3. **Stable JSON schema.** `--robot` luôn trả `{"schema":"openproxy.v1.<resource>","data":...,"meta":{...}}`. Schema version bump khi breaking.
4. **stdin/stdout là first-class.** Mọi `add/apply/import` chấp nhận `--from-file -` (đọc stdin). Mọi `list/get` có `--robot` để pipe.
5. **Exit codes có ý nghĩa.** `0` ok, `1` runtime error, `2` usage error (clap), `3` not found, `4` conflict, `5` auth, `6` network/server down, `7` validation.
6. **No interactive prompts in `--robot` mode.** Mọi input phải truyền qua flag hoặc stdin. TTY mode thì cho phép TUI/prompt.
7. **Long-running commands stream events.** `usage stream`, `logs tail`, `chat` — emit NDJSON khi `--robot`, mỗi dòng là một event.
8. **Self-documenting.** `openproxy <cmd> --help` đủ để agent dùng không cần web. `openproxy schema --resource provider` trả JSON Schema.
9. **Predictable side-effects.** Mỗi lệnh ghi DB phải log dòng `[mutated] <resource> <id>` tới stderr (trừ `--quiet`).
10. **Backward-compat.** Lệnh hiện tại (`provider list`, `key add`, `pool create`, …) giữ alias.

## 3. Kiến trúc
 
### 3.1 CLI vs Server — cùng binary, cùng DB
 
```
openproxy ──► main.rs
              │
              ├── if let Some(cmd) = cli.cmd { run_cli(cmd) }
              │     └── src/cli/  (mở rộng)
              │           ├── output.rs      ← Robot/Human formatter (--robot)
              │           ├── config.rs      ← ~/.config/openproxy/config.toml (optional)
              │           ├── runtime.rs     ← gọi server local cho các lệnh streaming
              │           └── commands/
              │                 ├── provider.rs   combo.rs   pool.rs
              │                 ├── key.rs        usage.rs   logs.rs
              │                 ├── mitm.rs       tunnel.rs  tool.rs
              │                 ├── translator.rs chat.rs    media.rs
              │                 ├── server.rs     settings.rs db.rs
              │                 ├── oauth.rs      doctor.rs  schema.rs
              │                 └── ...
              └── else { run_server() }  ← hiện tại
```
 
### 3.2 Tích hợp trực tiếp DB — server tự reload
 
Mọi lệnh state (`provider`, `combo`, `pool`, `key`, `settings`, …) chỉ làm 3 việc:
 
1. `Db::load()` từ `DATA_DIR`.
2. `db.update(|d| { ... })` mutate.
3. Exit.
 
Server (nếu đang chạy) phải auto-reload `ArcSwap<AppDb>` khi `db.json` thay đổi. **Đây là yêu cầu tiên quyết của M1**:
 
- Thêm dependency `notify = "7"` (file watcher).
- Trong `Db::load()` hoặc `AppState::new()`, spawn task watch `data_dir.join("db.json")` → reload và `arc_swap.store(...)` mỗi khi file thay đổi.
- Debounce 100ms để gộp nhiều ghi liên tục.
- Lệnh CLI ghi DB → trong vòng <200ms server đã thấy state mới. Không IPC, không API, không câu hỏi.
 
Lệnh **runtime-only** (cần subprocess hoặc state in-memory của server):
 
| Lệnh | Cần server vì |
|---|---|
| `logs tail`, `logs stats`, `logs clear` | Buffer in-memory + SSE stream |
| `usage stream` | SSE stream `/api/usage/stream` |
| `chat stream`, `chat send --stream` | Streaming completions qua server (route + RTK + fallback) |
| `mitm start/stop`, `mitm cert generate` | Subprocess do server quản |
| `tunnel start/stop`, `tunnel tailscale-*` | Subprocess (cloudflared/tailscaled) |
| `provider oauth start/poll/refresh` | Device flow callback handler trong server |
 
Nếu server không chạy → fail rõ ràng: `error: server not running. Start it: openproxy server start --detach` (exit 6).
 
### 3.3 Config & profile (optional, dùng khi remote-manage)
 
Mặc định CLI dùng `DATA_DIR` (env hoặc `--data-dir`) — không cần config file. Config file chỉ hữu ích khi:
 
- Quản lý nhiều instance OpenProxy ở các thư mục khác nhau.
- Quản lý server từ xa (remote management qua HTTPS).
 
`~/.config/openproxy/config.toml` (Linux/macOS), `%APPDATA%\openproxy\config.toml` (Windows):
 
```toml
default_profile = "local"
 
[profiles.local]
data_dir = "/home/u/.openproxy"
 
[profiles.remote]
url = "https://proxy.example.com"
api_key_env = "OPENPROXY_REMOTE_KEY"
```
 
CLI ưu tiên: flag `--data-dir`/`--url`/`--api-key` > `OPENPROXY_*` env > `--profile <name>` > `default_profile` > `$DATA_DIR` (default).
 
### 3.4 Output spec — `--robot`
 
Mọi lệnh `--robot`:
 
```json
{
  "schema": "openproxy.v1.provider.list",
  "ok": true,
  "data": [ ... ],
  "meta": { "count": 12, "duration_ms": 18, "online": true }
}
```
 
Lỗi:
```json
{
  "schema": "openproxy.v1.error",
  "ok": false,
  "error": { "code": "not_found", "message": "Provider 'xyz' not found" },
  "meta": { "duration_ms": 4 }
}
```
 
Streaming (NDJSON):
```
{"schema":"openproxy.v1.usage.event","data":{"ts":"...","tokens":42}}
{"schema":"openproxy.v1.usage.event","data":{"ts":"...","tokens":17}}
```

## 6. Agent UX — chi tiết
 
### 6.1 Khám phá lệnh không cần web
 
```
$ openproxy --robot schema list
{"schema":"openproxy.v1.schema.list","ok":true,"data":["provider","combo",...]}
 
$ openproxy --robot schema show provider
{"schema":"openproxy.v1.schema.provider","data":{"$schema":"...","type":"object",...}}
 
$ openproxy --robot schema example provider
{"schema":"openproxy.v1.example.provider","data":{"name":"openai-main","type":"openai","apiKey":"sk-..."}}
```
 
→ Agent đọc schema/example → tự sinh payload đúng → `provider apply --from-file -`.
 
### 6.2 Bulk / declarative
 
```
cat <<EOF | openproxy provider apply --from-file -
- name: openai-main
  type: openai
  apiKey: sk-xxx
  priority: 10
- name: groq-fallback
  type: groq
  apiKey: gsk-xxx
EOF
```
 
→ Trả về diff (`created`/`updated`/`unchanged`/`deleted` nếu `--prune`).
 
### 6.3 Idempotency
 
`provider apply` (default) = upsert. `provider add` = strict create (lỗi nếu trùng). `provider delete` = exit 0 nếu không tồn tại trừ khi `--strict`.
 
### 6.4 Streaming events
 
```
$ openproxy --robot logs tail --follow
{"schema":"openproxy.v1.log.event","data":{"ts":"...","level":"info","msg":"..."}}
{"schema":"openproxy.v1.log.event","data":{...}}
```
 
Agent đọc NDJSON từng dòng, dừng khi cần (`pkill openproxy`).
 
### 6.5 Stable contract
 
- `--robot` schema có version (`openproxy.v1.*`). Bump v2 khi breaking; v1 deprecated trong ≥ 2 minor releases trước khi xoá.
- Document trong `docs/cli-schema.md` (sinh tự động từ `schema show`).
 
### 6.6 Permissions cho agent (sandbox)
 
Đề xuất env `OPENPROXY_CLI_READONLY=1` → vô hiệu mọi lệnh write (cho agent untrusted chỉ đọc state). Hoặc API key có flag `readonly: true`.

## 7. Auth & secrets
 
- **Local mode (mặc định)**: CLI ghi/đọc DB trực tiếp → KHÔNG cần auth, KHÔNG cần API key, KHÔNG hỏi gì. Đơn giản.
- **Remote mode** (hiếm — quản lý server ở máy khác): chỉ kích hoạt khi user truyền `--url <https://...>`.
  - API key đọc theo thứ tự: `--api-key` → `OPENPROXY_API_KEY` → config profile (`api_key_env`/`api_key`).
  - Khi `requireLogin=true`: `openproxy auth login --url <u>` → POST `/api/auth/login` → lưu cookie vào `~/.config/openproxy/cookies` (`chmod 600`).
- Không bao giờ in API key vào stdout/stderr. Lệnh `key list` mặc định mask (`sk-abcd…1234`); `--show-secrets` để hiện full.

## 12. Câu hỏi cho user (xác nhận trước khi implement)
 
1. **Lệnh con vs flag**: bạn muốn `provider models alias set` (3 cấp) hay `provider model-alias set` (2 cấp gọn hơn)? PLAN này dùng 3 cấp cho rõ.
2. **DB watcher**: OK thêm `notify-rs` để server auto-reload khi CLI ghi DB? Đây là tiền đề cho mental model đơn giản (CLI = direct DB).
3. **Schema version**: bắt đầu thẳng `v1` (commitment cao) hay `v0` rồi `v1` khi đóng băng cuối M6?
4. **Bulk apply có `--prune`?** (xoá resource không có trong file, kubectl-style). Mặc định PLAN này: KHÔNG prune; bật `--prune` mới prune.
5. **Lệnh runtime khi server down**: fail rõ ràng (option A) hay tự động `server start --detach` rồi retry (option B)? PLAN này đề xuất A cho predictable, không magic.
6. **Phạm vi M1**: bắt đầu code luôn từ M1 sau khi bạn duyệt PLAN, hay muốn breakdown thêm?
