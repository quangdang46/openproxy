# OpenProxy ŌĆö Single-Binary Distribution Plan

Status: proposed
Owner: backend
Last updated: 2026-05-14

## 1. Goal

Ship `openproxy` as a single, cross-platform binary that:

1. Embeds the pre-built Astro dashboard (`web/dist/`) ŌĆö no build step at install or runtime.
2. Serves the dashboard, the OpenAI-compatible API (`/v1`), and the admin API (`/api`) from one process on one port (`4623`).
3. Installs in one command:
   ```bash
   curl -fsSL "https://raw.githubusercontent.com/quangdang46/openproxy/main/install.sh?$(date +%s)" | bash
   ```
4. After install, `openproxy` (no args) starts the server and auto-opens the browser to the UI.
5. Keeps the existing CLI (`openproxy provider ŌĆ”`, `combo ŌĆ”`, etc.) working unchanged.

## 2. Decisions

| # | Question | Decision |
|---|---|---|
| 1 | Sidecar handling | **1=b** Coexist behind a flag. Embedded assets are the default; `--dashboard-sidecar-url <url>` (or `DASHBOARD_SIDECAR_URL`) restores the current reverse-proxy behavior for UI development. |
| 2 | Release targets | **2=b** Four targets: `x86_64-unknown-linux-musl`, `aarch64-unknown-linux-musl`, `x86_64-apple-darwin`, `aarch64-apple-darwin`. Windows opt-in later (the install.sh already detects and exits cleanly). |
| 3 | `openproxy` no-args behavior | **3=a** Start the server on `4623` and auto-open the browser. Provide `--no-open` for headless/SSH/container contexts. |
| 4 | Release profile | **4=a** Standard: `lto = "fat"`, `codegen-units = 1`, `strip = "symbols"`, `panic = "abort"`, `opt-level = "s"`. |

## 4b. Phase 0b ŌĆö npm-package distribution (esbuild/biome pattern)

Status: Ō£ģ skeletons created; release workflow integration pending Phase 4.

Why: provide a second install channel ŌĆö `npm install -g openproxy` ŌĆö alongside the curl installer. Many users (Node-heavy stacks, CI configs, corporate environments) prefer npm. This gives parity with `esbuild`, `@biomejs/biome`, `swc`, `turbo`.

### Architecture: `optionalDependencies`, not `postinstall`

```
packages/
Ōö£ŌöĆŌöĆ openproxy/                    ŌåÉ meta package ŌĆö `npm install -g openproxy`
Ōöé   Ōö£ŌöĆŌöĆ package.json              ŌåÉ lists 4 platform packages as optionalDependencies
Ōöé   Ōö£ŌöĆŌöĆ bin/openproxy.js          ŌåÉ shim: resolves platform binary, execs, forwards args + exit code
Ōöé   ŌööŌöĆŌöĆ README.md                 ŌåÉ short, points to main README
ŌööŌöĆŌöĆ platform/
    Ōö£ŌöĆŌöĆ linux-x64/                ŌåÉ @openprx/openproxy-linux-x64
    Ōö£ŌöĆŌöĆ linux-arm64/              ŌåÉ @openprx/openproxy-linux-arm64
    Ōö£ŌöĆŌöĆ darwin-x64/               ŌåÉ @openprx/openproxy-darwin-x64
    ŌööŌöĆŌöĆ darwin-arm64/             ŌåÉ @openprx/openproxy-darwin-arm64
```

Each platform package's `package.json` has `"os"` + `"cpu"` constraints. npm/pnpm install only the matching one and silently skip the rest. The meta package's `bin/openproxy.js` shim does:
```js
require.resolve(`@openprx/openproxy-${platform}-${arch}/bin/openproxy`)
spawnSync(binPath, process.argv.slice(2), { stdio: 'inherit' })
process.exit(child.status ?? 1)
```

Why not `postinstall`? `postinstall` breaks under `--ignore-scripts`, can't run in offline mirrors, conflicts with Yarn PnP, and adds a security review surface. The optional-deps pattern is the modern standard.

### Release prerequisites (one-time, manual)

1. **Create npm org / scope `@openprx`:**
   ```bash
   npm login
   npm org create openprx
   ```
   Or skip the scope and use unscoped names (`openproxy-linux-x64`). Scoped is recommended ŌĆö clearer ownership, avoids name squatting.
2. **Add `NPM_TOKEN` to GitHub repo secrets** (`Settings ŌåÆ Secrets and variables ŌåÆ Actions ŌåÆ New repository secret`). Token must have publish permission on the `@openprx` scope.
3. **First-time publish flag:** scoped packages default to private. Publish each platform package once with `npm publish --access public` so they become publicly installable. The release workflow always passes `--access public` (idempotent).

### Publish ordering ŌĆö non-negotiable

```
1. Build all 4 binaries           (existing release.yml job)
2. Stage each binary into packages/platform/<target>/bin/openproxy
3. Publish all 4 @openprx/openproxy-* packages          ŌåÉ MUST come first
4. Wait ~30s for npm registry to propagate      ŌåÉ simple sleep is fine
5. Publish openproxy meta package
```

If meta is published first, users running `npm install -g openproxy` between step 5 and step 3 get an unresolved-optional-dep failure.

### Version policy

All 5 package.json files share the same version number, kept in lockstep with the GitHub release tag (`vX.Y.Z` ŌåÆ `X.Y.Z` for npm). The release workflow extracts the tag, rewrites all 5 `version` fields and the meta package's `optionalDependencies` versions before publishing.

### Workflow extension (sketch ŌĆö full version in Phase 4)

```yaml
publish-npm:
  needs: [release]   # only runs after binaries are attached to GitHub release
  runs-on: ubuntu-latest
  permissions:
    contents: read
  steps:
    - uses: actions/checkout@v4
    - uses: actions/setup-node@v4
      with:
        node-version: 20
        registry-url: https://registry.npmjs.org
    - name: Stage binaries from GitHub release
      run: node scripts/stage-npm-binaries.mjs "${GITHUB_REF_NAME}"
    - name: Publish platform packages
      env:
        NODE_AUTH_TOKEN: ${{ secrets.NPM_TOKEN }}
      run: |
        for d in packages/platform/*/; do
          (cd "$d" && npm publish --access public --provenance)
        done
    - name: Wait for registry propagation
      run: sleep 30
    - name: Publish meta package
      env:
        NODE_AUTH_TOKEN: ${{ secrets.NPM_TOKEN }}
      working-directory: packages/openproxy
      run: npm publish --access public --provenance
```

`--provenance` enables npm's signed provenance attestations, tying the published artifacts to the GitHub Actions run that produced them. Highly recommended for supply-chain visibility.

### 4.3 Rewrite `src/server/dashboard/mod.rs`

Replace the reverse-proxy fallback with an embed-backed handler. Keep the public `routes()` signature so `src/server/mod.rs` and `state.rs` need no changes elsewhere.

Behavior:
- Serve embedded `web/dist/<path>` for any request that falls through the API routers.
- For request paths without an extension (e.g. SPA routes like `/dashboard/usage`), serve `index.html` so the client router can take over.
- Add `Cache-Control: public, max-age=31536000, immutable` for hashed assets, `no-cache` for HTML.
- Emit ETag from the embedded file's sha256.
- If `dashboard_sidecar_url` is set on `AppState`, route fallback traffic through the existing reverse-proxy code path instead of the embed (development mode).

Pseudocode:
```rust
pub fn routes() -> Router<AppState> {
    Router::new().fallback(serve_dashboard)
}

async fn serve_dashboard(State(state): State<AppState>, req: Request<Body>) -> Response {
    if state.dashboard_sidecar_url.is_some() {
        return proxy_dashboard_request(State(state), req).await;  // existing code
    }
    serve_embedded(req.uri()).await
}
```

Concrete embed handler: see prior design (`rust-embed` + `mime_guess` + SPA fallback + ETag + cache headers).

### 4.5 CLI flag (`src/cli/server.rs` / `src/cli/mod.rs`)

## 6. Phase 2 ŌĆö Default `openproxy` to "start server + open browser" (3=a)

In `src/main.rs`, when `cli.cmd` is `None`:

1. Resolve port (`--port`, `OPENPROXY_PORT`, default `4623`).
2. Bind listener; if address is in use, exit with a clear error.
3. Spawn the server task.
4. Wait for a `/health` 200 (max ~3 s) before opening the browser.
5. If `--open` (default) and stdout is a TTY and `DISPLAY`/`WAYLAND_DISPLAY`/macOS, call `open::that("http://127.0.0.1:<port>/")`. On failure, log and continue.
6. Log:
   ```
   openproxy listening on http://127.0.0.1:4623
     Dashboard:  http://127.0.0.1:4623/
     OpenAI v1:  http://127.0.0.1:4623/v1
     Press Ctrl+C to stop.
   ```
7. Honor existing shutdown signals.

`--no-open` and non-TTY environments must not attempt to launch a browser.

## 7. Phase 3 ŌĆö CI workflow (`.github/workflows/ci.yml`)

**Status: Ō£ģ implemented.**

3-OS matrix (`ubuntu-latest`, `macos-latest`, `windows-latest`). Each job: pnpm-install ŌåÆ pnpm build ŌåÆ cargo fmt ŌåÆ cargo clippy (advisory) ŌåÆ cargo test.

The old `.github/workflows/rust-ci.yml` (Linux-only fmt+test) was deleted ŌĆö fully superseded.

### Trade-off accepted

The codebase has 115 pre-existing clippy warnings. Gating CI on `-D warnings` would block every PR until that's cleaned up ŌĆö out of scope for distribution work. The clippy step is currently advisory (compiles + surfaces lints, does not fail). Tighten to `-D warnings` after a separate cleanup pass.

The env-level `RUSTFLAGS: -D warnings` was removed as well ŌĆö it propagates to dependency builds and any warning in any dep would fail CI.

## 13. Risks and rollback

| Risk | Mitigation |
|---|---|
| Binary size grows beyond user tolerance | `compression` feature on `rust-embed`, exclude sourcemaps in Astro build, measure per-target size in CI; if >40 MB, revisit lazy-load model. |
| Astro build differs from Next sidecar in the current setup | Verify `web/dist/index.html` exists and routes match. The current Astro config already has `output: 'static'` so no migration needed. |
| Cross-compilation breaks for `aarch64-unknown-linux-musl` | `cross` handles this; pinned `cross` version in CI. |
| macOS Gatekeeper warns on unsigned binary | Document the `xattr -d com.apple.quarantine` workaround in README; defer notarization to a later phase. |
| Embedded handler regresses dashboard streaming endpoints | Streaming endpoints are under `/v1/*` and `/api/*` and are NOT served by the dashboard module ŌĆö they keep their existing handlers. The embed only handles the SPA fallback. |
| install.sh corrupts user dotfiles | `--easy-mode` is opt-in; default behavior prints the PATH line only. `--uninstall` removes the marker line. |

Rollback: revert the dashboard handler change behind the existing `--dashboard-sidecar-url` flag ŌĆö users can pass it (or set the env var) to restore the legacy sidecar setup until the regression is fixed.
